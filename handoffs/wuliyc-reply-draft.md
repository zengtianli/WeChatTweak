@wuliyc 你好，感谢分享 4.1.11 上「撤回提示 + 消息保留」可行的信息，这正是我们想做的变体。

背景同步：我们在 fork 里已支持 4.1.11（build 269136）的**静默防撤回**——把撤回 XML 解析器 `TryParseMessageXML`（`0x48a03b0` 处那条判断 msgType 是否为 `revokemsg` 的 `cbz`）改成无条件 `b`，直接跳过整个 revokemsg 解析分支。效果是 `newmsgid`/`replacemsg` 都不被解析，下游因此既不删消息也不弹提示，属于「静默保留」。

我们现在想做的正是你实现的「留提示」变体，思路是：**恢复解析（让下游拿到 newmsgid + replacemsg），只在下游 NOP 掉『按 newmsgid 删本地消息』的那次调用，保留插入撤回提示**。我们已逆向确认：删除动作**不在解析函数内**，而在消费解析结果的下游函数里，静态没能定位到那条删除调用的精确地址。

如果方便，能否请你分享一下你的具体改法：

1. 你改的是哪个函数 / 大致 VA（或符号名）？是解析器里、还是下游的消息管理 / DB 层？
2. 被改指令的**原始字节**是什么？
3. 改成了什么字节？
4. 对应 `wechat.dylib` 里哪一条指令（`bl` / `cbz` / `nop` 等）？

有这四项我们就能直接对上位置、快速验证。哪怕只是「改哪个函数附近、原字节大致长啥样」的方向性提示也很有帮助。
